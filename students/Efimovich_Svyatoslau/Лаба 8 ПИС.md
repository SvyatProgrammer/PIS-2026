# Лабораторная работа №8

**По дисциплине:** Проектирование интернет-систем
**Тема:** Микросервисы и Event Bus

---

## Цель работы

Разделить систему на независимые микросервисы (Bounded Contexts), реализовать асинхронное взаимодействие через брокер сообщений (**RabbitMQ**) и настроить единую точку входа через API Gateway.

---

## Вариант — Сервис заявок “Да, я в деле”

### 1. Выделение Bounded Contexts

Система разделена на два микросервиса:

* **Participation Service**
  Отвечает за создание заявок (`ParticipationRequest`), хранение и управление их статусами.

* **Event Service**
  Отвечает за управление событиями (мероприятиями), на которые пользователи могут откликаться.

Каждый сервис:

* имеет собственную базу данных
* развёрнут независимо
* не зависит напрямую от другого сервиса

---

### 2. Event Bus (RabbitMQ)

Для взаимодействия между сервисами используется брокер сообщений **RabbitMQ**.

**Сценарий:**

* При создании заявки в `Participation Service` генерируется событие `RequestCreatedEvent`
* Событие отправляется в Exchange
* `Event Service` подписан на очередь и получает это событие

**Назначение:**

* обновление информации о количестве участников
* запуск дополнительных процессов (уведомления и т.д.)

**Пример публикации события:**

```csharp
public class RabbitMqEventPublisher : IEventPublisher
{
    public Task Publish(object @event)
    {
        // сериализация и отправка в RabbitMQ
        Console.WriteLine($"Event sent: {@event.GetType().Name}");
        return Task.CompletedTask;
    }
}
```

---

### 3. API Gateway (Nginx)

В качестве API Gateway используется **Nginx**.

**Маршрутизация:**

* `/api/requests/**` → Participation Service
* `/api/events/**` → Event Service

**Пример конфигурации:**

```nginx
server {
    listen 80;

    location /api/requests/ {
        proxy_pass http://participation-service:5000/;
    }

    location /api/events/ {
        proxy_pass http://event-service:5001/;
    }
}
```

---

### 4. Docker Compose

Все сервисы объединены через Docker Compose:

* Participation Service
* Event Service
* PostgreSQL (для каждого сервиса)
* RabbitMQ
* Nginx

Это обеспечивает:

* удобный запуск всей системы
* изоляцию окружения
* простое масштабирование

---

### 5. Отказоустойчивость (Resilience)

Для повышения надёжности используется подход **Circuit Breaker** (концептуально):

* При недоступности `Event Service`
* `Participation Service` не ждёт ответа
* Событие остаётся в очереди и будет обработано позже

Это предотвращает:

* каскадные сбои
* перегрузку системы

## Контрольные вопросы

**1. Что такое bounded context?**
Это граница, внутри которой используется единая доменная модель. Каждый микросервис реализует свой bounded context.

---

**2. Почему микросервисы не должны делить БД?**
Общая база данных создаёт сильную связанность. Независимость сервисов достигается только при изолированных хранилищах.

---

**3. Зачем нужен Circuit Breaker?**
Он предотвращает каскадные сбои, позволяя системе быстро реагировать на недоступность зависимых сервисов.
