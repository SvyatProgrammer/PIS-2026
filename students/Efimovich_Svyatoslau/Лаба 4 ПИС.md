# Лабораторная работа №4

**По дисциплине:** Проектирование интернет-систем
**Тема:** Application Layer: Commands, Queries, Handlers

---

## Цель работы

Реализовать прикладной слой (Application Layer) системы на платформе **ASP.NET Core** с разделением операций на команды и запросы в соответствии с паттерном **CQRS**.

**Вариант — Сервис заявок “Да, я в деле”**

* Питч: Платформа, где пользователи могут откликаться на события/активности (например, участие в мероприятии).
* Ядро домена: агрегат `ParticipationRequest`
* Объекты-значения: `UserId`, `EventId`, `RequestStatus`

---

## Ход выполнения работы

### 1. Команды (Commands)

В прикладном слое реализованы команды как иммутабельные DTO (C# records), отвечающие за изменение состояния системы.

**`CreateParticipationRequestCommand`** — создание заявки на участие.

* Поля: `Guid UserId`, `Guid EventId`
* Назначение: инициирует создание новой заявки

**Пример:**

```csharp
public record CreateParticipationRequestCommand(Guid UserId, Guid EventId);
```

---

### 2. Command Handlers

Реализованы обработчики команд, инкапсулирующие бизнес-логику выполнения.

**`CreateParticipationRequestHandler`**

* Создаёт агрегат `ParticipationRequest`
* Сохраняет его через репозиторий
* Публикует доменные события

**Пример:**

```csharp
public class CreateParticipationRequestHandler
{
    private readonly IParticipationRequestRepository _repository;
    private readonly IEventPublisher _publisher;

    public CreateParticipationRequestHandler(
        IParticipationRequestRepository repository,
        IEventPublisher publisher)
    {
        _repository = repository;
        _publisher = publisher;
    }

    public async Task<Guid> Handle(CreateParticipationRequestCommand command)
    {
        var request = ParticipationRequest.Create(
            new UserId(command.UserId),
            new EventId(command.EventId)
        );

        await _repository.SaveAsync(request);

        foreach (var domainEvent in request.GetEvents())
        {
            await _publisher.Publish(domainEvent);
        }

        return request.Id;
    }
}
```

### 3. Запросы (Queries) и DTO

Для получения данных реализованы Query-объекты и DTO.

**`GetParticipationRequestQuery`** — получение заявки по ID

* Поле: `Guid RequestId`

**DTO: `ParticipationRequestResponse`**

* Поля:

  * `Id`
  * `UserId`
  * `EventId`
  * `Status`

**Пример:**

```csharp
public record ParticipationRequestResponse(
    Guid Id,
    Guid UserId,
    Guid EventId,
    string Status
);
```

---

### 4. Query Handlers

**`GetParticipationRequestHandler`**

* Получает данные из репозитория
* Преобразует доменную модель в DTO

**Пример:**

```csharp
public class GetParticipationRequestHandler
{
    private readonly IParticipationRequestRepository _repository;

    public GetParticipationRequestHandler(IParticipationRequestRepository repository)
    {
        _repository = repository;
    }

    public async Task<ParticipationRequestResponse> Handle(GetParticipationRequestQuery query)
    {
        var request = await _repository.GetByIdAsync(query.RequestId);

        if (request == null)
            throw new Exception("Request not found");

        return new ParticipationRequestResponse(
            request.Id,
            request.UserId.Value,
            request.EventId.Value,
            request.Status.ToString()
        );
    }
}
```

## Контрольные вопросы

**1. В чём разница между Command и Query?**
Command изменяет состояние системы, а Query только читает данные без изменений.

---

**2. Почему Command Handler возвращает только ID?**
Это уменьшает связанность и позволяет получать актуальные данные через отдельный Query.

---

**3. Где выполняется валидация?**

* Формат данных — в слое Application
* Бизнес-правила — в доменной модели

---

**4. Можно ли вызывать Query из Command Handler?**
Не рекомендуется, так как нарушает принцип CQRS. Лучше работать напрямую с репозиториями.

---

**5. Зачем разделять Request DTO и Command?**
Чтобы отделить внешний API от внутренней логики и сделать систему гибкой к изменениям.
