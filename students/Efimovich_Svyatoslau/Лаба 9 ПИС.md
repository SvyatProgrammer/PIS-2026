## Вариант — Сервис заявок “Да, я в деле”

### 1. Протофайлы (.proto)

Разработан файл `participation.proto`, описывающий сервис `ParticipationGrpcService`.

Контракт включает:

* описание сообщений (Request/Response)
* определение методов сервиса

**Пример:**

```proto
syntax = "proto3";

option csharp_namespace = "GrpcParticipation";

service ParticipationGrpcService {
  rpc GetRequest (GetRequestById) returns (ParticipationResponse);
  rpc StreamUserRequests (UserRequest) returns (stream ParticipationResponse);
}

message GetRequestById {
  string id = 1;
}

message UserRequest {
  string userId = 1;
}

message ParticipationResponse {
  string id = 1;
  string userId = 2;
  string eventId = 3;
  string status = 4;
}
```

Использование Protocol Buffers обеспечивает:

* строгую типизацию
* компактную бинарную сериализацию
* автоматическую генерацию кода

### 2. gRPC Server

Реализован сервер на **ASP.NET Core gRPC**.

Поддерживаются два типа вызовов:

#### Unary RPC — `GetRequest`

Обычный запрос-ответ:

* клиент отправляет ID заявки
* сервер возвращает данные

#### Server Streaming — `StreamUserRequests`

Позволяет:

* получать список заявок пользователя
* обрабатывать поток данных в реальном времени

**Пример:**

```csharp
public class ParticipationGrpcService : ParticipationGrpcService.ParticipationGrpcServiceBase
{
    private readonly IParticipationRequestRepository _repository;

    public override async Task<ParticipationResponse> GetRequest(
        GetRequestById request,
        ServerCallContext context)
    {
        var entity = await _repository.GetByIdAsync(Guid.Parse(request.Id));

        return new ParticipationResponse
        {
            Id = entity.Id.ToString(),
            UserId = entity.UserId.Value.ToString(),
            EventId = entity.EventId.Value.ToString(),
            Status = entity.Status.ToString()
        };
    }

    public override async Task StreamUserRequests(
        UserRequest request,
        IServerStreamWriter<ParticipationResponse> responseStream,
        ServerCallContext context)
    {
        var userId = Guid.Parse(request.UserId);
        var requests = await _repository.GetByUserIdAsync(userId);

        foreach (var r in requests)
        {
            await responseStream.WriteAsync(new ParticipationResponse
            {
                Id = r.Id.ToString(),
                UserId = r.UserId.Value.ToString(),
                EventId = r.EventId.Value.ToString(),
                Status = r.Status.ToString()
            });
        }
    }
}
```

### 3. gRPC Client

Клиент использует автоматически сгенерированные **stubs**.

**Возможности:**

* вызов Unary методов
* чтение потоков данных (streaming)

Это позволяет легко интегрировать сервисы между собой.

---

### 4. Сравнение gRPC и REST

В ходе работы выявлены преимущества gRPC:

* бинарный формат (Protobuf) → меньше размер данных
* HTTP/2 → мультиплексирование
* встроенный streaming
* генерация клиентского кода

## Контрольные вопросы

**1. В чём преимущество gRPC над REST?**
gRPC использует HTTP/2 и бинарную передачу данных, что делает его быстрее и эффективнее по сравнению с REST.

---

**2. Почему Protocol Buffers быстрее JSON?**
Потому что это бинарный формат с заранее известной структурой, что исключает необходимость сложного парсинга текста.

---

**3. Зачем нужен streaming в gRPC?**
Он позволяет передавать данные в реальном времени без постоянных HTTP-запросов, что снижает нагрузку и ускоряет взаимодействие.
