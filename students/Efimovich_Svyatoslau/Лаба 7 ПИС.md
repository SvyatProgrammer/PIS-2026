## Цель работы

Реализовать паттерн CQRS, разделив доменную модель (Write Model) и модель представления (Read Model), обеспечив их синхронизацию через события.

---

## Вариант — Сервис заявок “Да, я в деле”

### 1. Write Model

В качестве модели записи используется агрегат `ParticipationRequest`.

Он отвечает за:

* обработку команд (создание заявки)
* проверку инвариантов (валидность `UserId`, `EventId`)
* генерацию доменных событий (`RequestCreatedEvent`)

Данные хранятся в нормализованном виде в таблице `participation_requests`.

---

### 2. Read Model

Создана отдельная таблица `participation_request_view`.

Она является денормализованной и содержит:

* `Id`
* `UserId`
* `EventId`
* `Status`
* `CreatedAt`

В отличие от Write Model, данные уже подготовлены для отображения и не требуют дополнительных преобразований.

---

### 3. Event-Driven Synchronization

Для синхронизации используется проекция `ParticipationRequestProjection`.

При возникновении события `RequestCreatedEvent`:

* создаётся запись в Read Model
* данные преобразуются в удобный для чтения формат

**Пример:**

```csharp
public class ParticipationRequestProjection
{
    private readonly ReadDbContext _context;

    public ParticipationRequestProjection(ReadDbContext context)
    {
        _context = context;
    }

    public async Task Handle(RequestCreatedEvent @event)
    {
        var view = new ParticipationRequestView
        {
            Id = @event.RequestId,
            UserId = @event.UserId,
            EventId = @event.EventId,
            Status = "Pending",
            CreatedAt = DateTime.UtcNow
        };

        _context.RequestViews.Add(view);
        await _context.SaveChangesAsync();
    }
}
```

Синхронизация может выполняться:

* синхронно (в рамках транзакции)
* асинхронно (через очередь сообщений)

---

### 4. Оптимизация запросов

Для ускорения чтения добавлены индексы:

* по `UserId` — для получения заявок пользователя
* по `CreatedAt` — для сортировки по дате

Это позволяет эффективно реализовать:

* просмотр заявок пользователя
* отображение последних активностей

## Контрольные вопросы

**1. В чём разница между CQRS и CQS?**
CQS — это принцип разделения методов (Command и Query) внутри класса.
CQRS — это архитектурный подход, при котором полностью разделяются модели записи и чтения.

---

**2. Почему Read Model денормализованная?**
Потому что она оптимизирована под чтение. Денормализация позволяет избежать сложных JOIN и ускоряет выполнение запросов.

---

**3. Что такое Eventual Consistency?**
Это состояние системы, при котором Read Model может временно отставать от Write Model, но со временем синхронизируется через обработку событий.

Реализация CQRS позволила разделить ответственность между записью и чтением данных.
